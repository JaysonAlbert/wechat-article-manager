# 买了 RTX 5090 之后可以干什么

拿到 **RTX 5090 32GB（不带D）** 第一反应是什么？打游戏当然可以，但我选择了另一条路——**本地部署大语言模型，构建 AI 编程工作**。

## 为什么选择本地 LLM

- **隐私安全**：代码和数据不出本地
- **无成本限制**：没有 API token、调用次数限制
- **完全自定义**：可以调整模型参数、上下文长度
- **离线可用**：不需要联网也能编程

## Ollama：本地 LLM 的事实标准

### 安装流程

```bash
# 1. 下载并安装 Ollama
# Windows: https://ollama.com/download
# 或
winget install Ollama

# 2. 拉取模型
ollama pull qwen3.5:35b

# 3. 验证安装
ollama run qwen3.5:25b "你好，请介绍一下你自己"
```

### 模型选择：性能测试

我从 **qwen3.5:35B**、**gemma4:26B**、**gemma4:31B** 三个模型中进行了深入测试，对比在不同上下文长度下的：

- Tokens/秒 (速度)
- VRAM 显存占用

测试数据详见：[LLM Context Length Performance Benchmark](https://github.com/JaysonAlbert/llm-profile/wiki/Index.md)

#### 推荐指数对比

评分标准：**综合考虑速度、上下文长度、效果**
- ⭐⭐⭐⭐⭐：速度稳定 + 超长上下文 + 效果好
- ⭐⭐⭐⭐：速度快 + 中等上下文 + 效果好
- ⭐⭐⭐：速度快 + 上下文短或 VRAM 占用高
- ⭐⭐：速度一般 + VRAM 占用高
- ⭐：速度太慢

**最终选择**：`qwen3.5:35B`

理由：
- **速度稳定**：不同上下文长度下保持 150+ tokens/s
- **显存友好**：相比 31B 模型，显存占用更可控
- **上下文超长**：原生支持 256k 上下文
- **效果好**：逻辑推理和代码生成能力出色

## OpenCode：黑盒 vs 白盒的选择

### 三个选项对比

| 工具 | 特点 | 问题 |
|------|------|------|
| **Claude Code** | 功能强大，API 集成 | 🚫 黑盒，看不到思考过程<br>🚫 经常卡住，GPU 利用率低 |
| **Codex** | OpenAI 出品 | ⚪ 未测试，暂不评价 |
| **OpenCode** | 开源透明，可配置 | ✅ 白盒，完整思考过程 |

### 为什么选择 OpenCode

**1. 可观测性**
- 可以看到 AI 的思考过程
- 知道什么时候在"思考"，什么时候在"卡住"
- 不再焦虑 GPU 是否卡住

**2. 可配置性**
- 可以手动设置上下文长度
- **关键**：右上角 Context 会显示当前上下文百分比
- 性能可测试，可调整

**3. 开源透明**
- 代码开源，可以审查
- 社区活跃，问题响应快

## 配置指南

### 设置上下文长度（关键！）

```bash
# 启动 OpenCode 时指定上下文长度
opencode --context-length 65536
```

**为什么需要手动设置？**
- 默认值通常不会显示进度
- 设置后，右上角会显示：`Context: 45%`
- 实时了解 token 使用情况

### 最佳实践配置

```bash
ollama run qwen3.5:35b
opencode --model qwen3.5:35b --context-length 65536
```

## 性能数据参考

根据我的测试，**qwen3.5:35B** 在不同上下文长度下的表现：

### Context 32k
- Token/s: **157.24** (解码速度)
- 预填充：**3438.68** token/s
- 适合中等规模代码库

### Context 64k
- Token/s: **159.02** (解码速度)
- 预填充：**3362.11** token/s
- **性价比最高**：速度与能力平衡

### Context 256k
- Token/s: **45.03** (解码速度)
- 预填充：**2537.96** token/s
- 适合超大代码库，速度略有下降
